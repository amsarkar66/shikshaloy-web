06
Identified Gaps
  • No Supabase database schema for school data Tables for students, staff, classes, fees, attendance, etc. have not been created. The entire admin module depends on mock data files that must be replaced with real queries before any production use.
  • No Row-Level Security (RLS) policies Without RLS, an admin user could potentially read data from other schools once DB queries are wired. School-scoped policies must be written before data is made live.
  • Sub-route access control not enforced Any logged-in user (e.g. a teacher or parent) who knows the URL /dashboard/fees can access it directly. Route-level role checks are missing from these pages.
  • CRUD actions are UI-only stubs Add Student, Collect Fee, Mark Attendance, Edit Record, Delete — all render UI but do nothing. No Server Actions or API routes have been implemented for any admin write operation.
  • Quick Actions panel is non-functional The 4 quick-action buttons on the overview dashboard are plain <button> elements with no onClick handlers or routing.
  • Transport module not linked to the Driver role The admin can view routes and buses, but driver assignment and the driver’s own route view are not yet connected. The driver role seed script exists but data is not shared.
  • Download / Print features are stubs Reports, Payroll, and Fee pages show Download/Print buttons that do not generate or fetch any files. Supabase Storage is not configured for admin-generated documents.
  • No real-time data (Messages / Activity Feed) The Messages module and the Recent Activity feed are static mock. Supabase Realtime channels are not subscribed. This is expected at this stage but must be addressed before beta.
  • Analytics charts are static arrays Enrolment, fee, and attendance trend charts use hardcoded monthly arrays. Period selector (This Term / 2025-26) switches between in-memory objects, not DB-aggregated data.
  • No notification / alert system The overview flags pending dues and overdue items visually, but there is no push notification, email, or in-app alert pipeline wired up for the admin role.
07
Recommendations & Next Steps
1
Build the Supabase schema (P0 — blocker)

Create tables for schools, students, staff, classes, subjects, attendance, fees, expenses, payroll. Add a school_id foreign key on every tenant-scoped table. Run migrations via supabase apply_migration.

2
Write RLS policies before wiring any queries (P0 — security)

Every table must have a policy: auth.uid() = admin_id AND school_id = (SELECT school_id FROM admin_profiles WHERE id = auth.uid()). Do not expose DB queries without this in place.

3
Add sub-route role guards (P0 — security)

Extend middleware.ts to check user_metadata.role against allowed roles for each route prefix (e.g. /dashboard/fees → admin, staff only). Redirect unauthorised users to their own overview.

4
Prioritise the three highest-value modules for DB integration first

Recommend: Students (most referenced), Attendance (daily workflow), Fee Management (revenue-critical). Wire real queries + Server Actions for these three before moving on to facilities modules.

5
Wire Quick Actions to real destinations

The four overview quick-action buttons should navigate to the correct module page (or open a modal). “Add Student” → /dashboard/students?new=1, “Mark Attendance” → /dashboard/attendance, etc. Low effort, high perceived completion.

6
Connect Transport module to the Driver role

When a driver is assigned to a route in admin’s Transport view, that route should appear in the driver’s sidebar view. This requires a shared routes table with a driver_id FK and RLS policies for both roles.

7
Replace analytics static arrays with aggregated DB queries

Analytics and the overview KPI cards should pull from real data. Consider Supabase database functions (RPCs) or materialised views for performance (e.g. monthly_fee_summary, daily_attendance_rate).

8
Add Supabase Storage buckets for downloadable documents

Reports, payslips, and student records will need file storage. Create private buckets (school-reports, payslips) with signed URL access scoped to admin’s school.
